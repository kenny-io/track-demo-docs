Pulsekit is a tiny heartbeat and uptime monitoring toolkit for Node.js services. Each service reports heartbeats to a shared monitor; a service that misses its interval is considered down.

## Install

```bash
npm install pulsekit
```

## Quickstart

```js
import { createMonitor } from "pulsekit";

const monitor = createMonitor({ intervalMs: 30_000, watchIntervalMs: 5_000 });

monitor.watch(({ name, from, to, at }) => {
  if (to === "down") page(`${name} stopped beating`);
});

// Somewhere in each service's loop:
monitor.beat("payments-api");
monitor.beat("nightly-batch", { intervalMs: 90 * 60_000 }); // its own cadence

// Later:
monitor.status("payments-api"); // "up" | "down" | "unknown"
monitor.summary(); // { up: 2, down: 0, total: 2 }
```

## How it works

- A **monitor** tracks named services in memory.
- Calling `beat(name)` registers the service on first call and refreshes its last-seen timestamp on every call.
- A service whose last heartbeat is older than `intervalMs` reports as `"down"`; unregistered services report `"unknown"`.
- Status is evaluated lazily on read, so an idle monitor costs nothing.
- **Transition watchers** let you opt into push-style alerts: set `watchIntervalMs` when creating the monitor, then call `watch(listener)` to receive `{ name, from, to, at }` events.
- Each service can have its own interval via the `beatOptions` parameter.

## When to use it

Pulsekit answers "is every part of my process still alive?" from inside the
process: cron loops, queue workers, socket consumers. It is in-memory and
per-process by design — pair it with an external probe for whole-host
failures, and use `toJSON()`/`restore()` to keep history across planned
restarts.

See the [API reference](/pulsekit/api) for the full surface.
